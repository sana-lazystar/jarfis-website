# Guides & Customization

실제 프로젝트에서 JARFIS를 최대한 활용하는 실용적인 패턴 모음.

## Meeting-to-Work 연결

킥오프 미팅을 워크 세션의 입력 컨텍스트로 활용하는 것은 JARFIS의 가장 강력한 패턴 중 하나입니다. 계획 단계의 결정이 구현 단계까지 일관되게 흐릅니다.

### Step 1: 킥오프 미팅 진행

```
/jarfis:meeting Define the authentication strategy for the mobile app
```

미팅은 구조화된 결정과 산출물을 미팅 디렉토리에 저장합니다.

### Step 2: 미팅 참조와 함께 작업 시작

```
/jarfis:work Implement mobile auth with biometric login --meeting auth-strategy-meeting
```

`--meeting` 플래그는 미팅 출력물을 입력 컨텍스트로 연결합니다. 상태 파일에 `meeting_ref`와 `meeting_dir` 필드가 포함되며, Agent들은 워크플로우 전반에 걸쳐 미팅 결정을 참조합니다.

이 패턴은 구현 전에 여러 이해관계자가 합의해야 하는 복잡한 기능에서 특히 가치 있습니다.

## `required_roles`를 통한 역할 설정

모든 프로젝트에 11개 Agent 전부가 필요하지는 않습니다. JARFIS는 `.jarfis-state.json`의 `required_roles` 필드로 어떤 Agent를 활성화할지 제어합니다:

```json
{
  "required_roles": {
    "backend": true,
    "frontend": true,
    "ux": false,
    "devops": false,
    "security": true
  }
}
```

### 일반적인 역할 패턴

**프론트엔드 전용 프로젝트** (예: 랜딩 페이지 리디자인):
```json
{
  "required_roles": {
    "backend": false,
    "frontend": true,
    "ux": true,
    "devops": false,
    "security": false
  }
}
```
Phase 3 (UX Design)이 활성화됩니다. Phase 4는 Frontend Engineer만 실행합니다. Backend와 DevOps Agent는 호출되지 않습니다.

**보안 요구사항이 있는 풀스택 프로젝트**:
```json
{
  "required_roles": {
    "backend": true,
    "frontend": true,
    "ux": true,
    "devops": true,
    "security": true
  }
}
```
모든 Phase가 활성화됩니다. Phase 4는 BE, FE, DevOps가 병렬로 실행됩니다. Phase 5에 Security Engineer 리뷰가 포함됩니다.

**백엔드 API 전용 프로젝트**:
```json
{
  "required_roles": {
    "backend": true,
    "frontend": false,
    "ux": false,
    "devops": true,
    "security": true
  }
}
```
Phase 3 (UX Design)이 완전히 건너뛰어집니다. Phase 4는 Backend와 DevOps만 실행합니다.

JARFIS는 Phase T (Triage) 동안 이 역할들을 자동으로 결정하지만, 필요에 따라 조정할 수 있습니다.

## 완료된 워크플로우 이어서 진행

완료된 워크플로우에 추가 작업이나 수정이 필요할 때는 처음부터 다시 시작하는 대신 `/jarfis:continue`를 사용하세요. 기존 브랜치와 산출물을 재활용합니다:

```
/jarfis:continue Add unit tests for the auth module
```

`--mode` 플래그를 사용하면 자동 분류를 건너뛸 수 있습니다:

```
/jarfis:continue Fix the token refresh edge case --mode fix
```

불필요한 Triage/Discovery Phase를 건너뛰기 때문에 후속 작업에서 `/jarfis:work`보다 빠릅니다.

## 프로젝트 프로파일

프로젝트 프로파일은 JARFIS가 코드베이스 구조와 컨벤션을 깊이 이해하도록 도와줍니다.

### 프로파일 초기화

```
/jarfis:project-init
```

프로젝트를 분석하여 `.jarfis/project-profile.md`를 생성합니다. 다음 정보를 포함합니다:
- 저장소 구조와 조직
- 의존성과 버전
- 빌드 도구와 스크립트
- 감지된 컨벤션과 패턴

### 프로파일 최신 상태 유지

프로젝트 구조가 크게 변경된 경우 (새 패키지 추가, 주요 의존성 업데이트 등):

```
/jarfis:project-update
```

수동으로 추가한 컨텍스트를 유지하면서 프로파일을 갱신합니다.

## Gate 활용

Gate는 필수 인간 체크포인트입니다. 효과적으로 활용하는 방법을 소개합니다.

### Gate 1 — Discovery 이후

Gate 1에서는 PRD와 타당성 평가를 검토합니다. 선택지:

- **승인**: Architecture & Planning (Phase 2)으로 진행
- **수정**: 구체적인 피드백 제공. PO와 Architect가 출력물을 수정하고 재발표합니다
- **중단**: 워크플로우 전체 취소

**팁**: 방향을 바꾸기에 가장 비용이 적은 지점입니다. Gate 1에서의 수정은 몇 분이면 충분하지만, Gate 3에서의 수정은 몇 시간이 걸릴 수 있습니다.

### Gate 2 — Architecture & UX 이후

Gate 2에서는 아키텍처 설계, 태스크 분해, (해당되는 경우) UX 명세를 검토합니다. 선택지:

- **승인**: Implementation (Phase 4)으로 진행
- **수정**: 아키텍처, 태스크 범위, UX 디자인 변경 요청
- **중단**: 워크플로우 취소

**팁**: `tasks.md`를 꼼꼼히 확인하세요 — 이 파일이 정확히 무엇을 구현할지 정의합니다. 여기서 누락된 태스크는 나중에 누락된 기능이 됩니다.

### Gate 3 — Review & QA 이후

Gate 3에서는 완성된 구현, 테스트 결과, 보안 발견 사항을 검토합니다. 선택지:

- **승인**: Retrospective (Phase 6)으로 진행 후 마무리
- **수정 후 재검토**: 수정 요청 후 리뷰 사이클 재시작
- **중단**: 워크플로우 취소
- **설계 재검토**: 리뷰 중 발견된 근본적인 설계 문제(병리적 패턴)의 경우 Phase 2로 복귀

## 학습 시스템 관리

JARFIS의 학습 시스템은 매 워크플로우를 이전보다 더 나아지게 하는 핵심입니다.

### 전역 학습: `~/.claude/jarfis-learnings.md`

홈 디렉토리에 위치하며 모든 프로젝트에서 공유됩니다. 다음을 포함합니다:

- **Agent Hints**: "새 미들웨어를 만들기 전에 항상 기존 미들웨어를 확인하라"나 "mutateAsync는 try-catch로 감싸 unhandled rejection을 방지하라" 같은 행동 규칙
- **Workflow Patterns**: "모노레포 프로젝트에서는 패키지별로 영향 분석을 실행하라" 같은 검증된 접근 방식

이 파일은 모든 워크플로우의 Phase 0에서 로드되고 Phase 6 (Retrospective)에서 갱신됩니다.

### 프로젝트 컨텍스트: `.jarfis/context.md`

워크플로우 전반에 걸쳐 Agent가 참조하는 프로젝트별 지식:

```markdown
# Project Context

## Tech Stack
- Frontend: Next.js 15 (App Router), TypeScript strict, Tailwind CSS 4
- Backend: FastAPI (Python 3.12), PostgreSQL 16, Redis
- Infrastructure: AWS ECS, Terraform, GitHub Actions

## Conventions
- API responses follow RFC 9457 Problem Details format
- All database models use UUID primary keys
- Frontend components follow atomic design
- Commit format: conventional commits (feat/fix/chore)

## Business Domain
- SaaS B2B platform for legal document management
- Compliance: SOC 2 Type II, GDPR

## Integration Constraints
- Auth: Auth0 (cannot change)
- Payment: Stripe (webhooks configured)
- Storage: S3 with presigned URLs
```

`context.md`가 구체적일수록 Agent 출력물의 정확도가 높아집니다. 워크플로우 시작 전에 투자할 수 있는 가장 높은 레버리지를 가진 작업입니다.

### `/jarfis:upgrade`로 학습 정리

시간이 지나면 `jarfis-learnings.md`에 오래됐거나 중복된 항목이 쌓일 수 있습니다. 업그레이드 명령어로 검토하고 정리하세요:

```
/jarfis:upgrade
```

기존 학습 항목을 하나씩 살펴보며 유지, 편집, 삭제할 수 있습니다.

## 컨텍스트 압축 후 워크플로우 복구

장기간 실행되는 워크플로우는 Claude Code의 컨텍스트 압축을 만날 수 있습니다. JARFIS는 이를 복구하도록 설계되었습니다:

1. `.jarfis-state.json` 파일이 정확한 Phase와 진행 상태를 유지합니다
2. `last_checkpoint` 필드에 압축 전 진행 상황이 기록됩니다
3. 디스크의 학습 파일과 산출물은 컨텍스트 압축의 영향을 받지 않습니다

컨텍스트 초기화 후 재개하려면 동일한 설명으로 `/jarfis:work`를 다시 실행하세요. JARFIS가 기존 `.jarfis-state.json`을 감지하고 처음부터 시작하는 대신 마지막 체크포인트에서 재개합니다.

## JARFIS 자체 수정

JARFIS의 동작을 커스터마이징해야 할 때 — 새 명령어 추가, Agent 프롬프트 수정, 워크플로우 확장 — 전용 자체 수정 명령어를 사용하세요:

```
/jarfis:implement Add a /jarfis:standup command for daily status reports
```

이 명령어는 JARFIS 시스템 구조를 이해하고 영향받는 모든 파일에 변경 사항이 올바르게 적용되도록 보장합니다.

## 실용 예시

### 예시 1: 프론트엔드 전용 기능

```
/jarfis:work Redesign the dashboard with new chart components
```

JARFIS가 `required_roles.backend`를 false로 설정하고 백엔드 관련 작업을 건너뜁니다. Phase 3 (UX Design)이 활성화되고 Phase 4는 Frontend Engineer만 호출합니다.

### 예시 2: 미팅 입력을 활용한 풀스택 작업

```
/jarfis:meeting Plan the multi-tenant architecture migration
/jarfis:work Implement multi-tenant data isolation --meeting tenant-architecture
```

미팅 결정 사항(예: "PostgreSQL에서 row-level security 사용")이 워크 세션으로 흘러들어가, Architect와 Backend Engineer가 합의된 접근 방식을 따르도록 보장합니다.

### 예시 3: 최소한의 오버헤드로 빠른 수정

```
/jarfis:work Fix the broken pagination on the orders list
```

단순한 버그 수정의 경우 JARFIS의 Triage Phase (Phase T)가 이를 Type C (단순 수정)로 분류하고 Phase 파이프라인을 간소화합니다. UX Design이나 Operational Readiness 같은 무거운 Phase를 건너뜁니다.

### 예시 4: 완료된 작업 이어서 진행

```
/jarfis:continue Add internationalization support to the auth module
```

이전 auth 워크플로우를 이어서 진행하며, 브랜치와 기존 산출물을 재활용합니다. 밀접하게 관련된 작업에서 새 워크플로우를 시작하는 것보다 빠릅니다.

### 예시 5: 문제 진단

```
/jarfis:health
```

워크플로우가 멈추거나 예상치 못한 동작을 보이는 경우, health 명령어로 좀비 프로세스와 기타 운영 이슈를 진단합니다.
