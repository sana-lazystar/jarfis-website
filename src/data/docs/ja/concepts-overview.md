# アーキテクチャ＆概念

JARFISが9人の専門AIエージェントを調整してプロダクションソフトウェアを提供する方法。

## オーケストレーションモデル

JARFISは**Claude Codeをオーケストレーター**として構築されています。単一エージェントシステムとは異なり、JARFISは9人の専門エージェントを — それぞれ独自の役割を持つ — 構造化されたフェーズパイプラインを通じて調整します。

## 9人のエージェントの役割

| エージェント | 主要な責務 | 主要アーティファクト |
|------------|-----------|-----------------|
| **Product Owner (PO)** | 要件定義、ユーザーストーリー作成、バックログ優先順位付け | `press-release.md`, `prd.md` |
| **Architect** | システム設計、技術決定、ADRドキュメント化 | `architecture.md`, `impact-analysis.md` |
| **Tech Lead** | コード標準、PRレビュー基準、エージェント間調整 | `tasks.md`, `review.md` |
| **UX Designer** | ユーザーフロー、ワイヤーフレーム、コンポーネント仕様 | `ux-spec.md` |
| **BE Engineer** | API実装、データモデル、バックエンドサービス | `api-spec.md`、バックエンドコード |
| **FE Engineer** | UIコンポーネント、状態管理、フロントエンド統合 | フロントエンドコード |
| **DevOps/SRE** | インフラコード、CI/CDパイプライン、デプロイ | `deployment-plan.md` |
| **QA Engineer** | テスト戦略、テストケース作成、品質ゲート | `test-strategy.md` |
| **Security Engineer** | 脅威モデリング、脆弱性レビュー、コンプライアンス | `review.md`内のセキュリティレビュー |

## フェーズパイプライン（8以上のフェーズ）

JARFISはすべての機能デリバリーを定義されたフェーズを通じて構造化します：

| フェーズ | 名前 | 目的 | 主要エージェント |
|---------|------|------|--------------|
| T | キックオフ | 機能スコープの確立と要件収集 | PO, Architect |
| 0 | 基盤整備 | 技術とアーキテクチャの決定 | Architect, Tech Lead |
| 1 | デザイン | UX仕様とAPI契約定義 | UX Designer, Architect, BE |
| 2 | アーキテクチャ | 完全なシステム設計 | Architect, BE, FE |
| 3 | 実装 | 並行開発 | BE, FE |
| 4 | 統合 | API連携、E2E統合 | BE, FE, Tech Lead |
| 4.5 | QA＆セキュリティ | テスト実行、セキュリティ監査 | QA, Security |
| 5 | デプロイ | プロダクションデプロイ | DevOps, Tech Lead |
| 6 | レトロスペクティブ | 学習の記録 | 全エージェント |

## 11のアーティファクト

すべてのJARFISワークフローは`.jarfis/works/{日付}/{機能名}/`に最大11の構造化アーティファクトを生成します：

| アーティファクト | 目的 |
|--------------|------|
| `press-release.md` | 顧客言語での機能発表文 |
| `prd.md` | プロダクト要件ドキュメント |
| `impact-analysis.md` | 技術的リスクとスコープ分析 |
| `architecture.md` | システム設計、コンポーネント図、ADR |
| `api-spec.md` | API契約（エンドポイント、型、認証） |
| `tasks.md` | フェーズ別開発タスク分解 |
| `test-strategy.md` | テスト計画、カバレッジ目標 |
| `ux-spec.md` | ワイヤーフレーム、コンポーネント仕様 |
| `deployment-plan.md` | リリース計画、ロールバック戦略 |
| `review.md` | 実装後のコード＆セキュリティレビュー |
| `retrospective.md` | フェーズ学習と改善アクション |

## 状態管理: `.jarfis-state.json`

JARFISはプロジェクトルートの`.jarfis-state.json`でワークフロー状態を管理します。このファイルはすべてのClaude Codeセッション開始時に**PreCompactフック**によって読み込まれ、正確なフェーズとコンテキストを復元します。

## PreCompactフック

PreCompactフックはコンテキスト圧縮前に実行されるClaude Codeライフサイクル統合です：

1. `.jarfis-state.json`から現在のフェーズとエージェント状態を**読み込みます**
2. 関連アーティファクトの要約を圧縮されたコンテキストに**注入します**
3. `jarfis-learnings.md`の重要な決定事項と学習を**保存します**
4. 現在のフェーズのアクティブなエージェントの視点を**復元します**

## 学習システム

JARFISは2つの学習メカニズムを通じて継続的に改善されます：

**`jarfis-learnings.md`**: フェーズ6レトロスペクティブ後に更新されるプロジェクトレベルの知識ベース。

**`context.md`**: すべてのエージェントプロンプトに注入されるプロジェクト固有のコンテキスト。

## ミーティングシステム (`/jarfis:meeting`)

エージェントが同期する必要がある場合 — ブロッカーの解決、共同決定、スコープの調整 — 構造化されたミーティングを開催できます：

```
/jarfis:meeting
```

## 設計原則

1. **専門化** — 各エージェントは1つのドメインで深い専門知識を持ちます
2. **アーティファクト駆動** — すべてのフェーズは人間が読めるレビュー可能な成果物を生成します
3. **フェーズゲート** — 前のフェーズのアーティファクトなしに次のフェーズは開始されません
4. **状態保持** — `.jarfis-state.json`を通じてセッション境界を越えて作業が継続されます
5. **学習** — すべてのプロジェクトが学習システムを通じて次のプロジェクトを速くします
6. **Claude Codeネイティブ** — 外部サービスなしで開発環境内で完全に実行されます
